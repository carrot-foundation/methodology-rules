# Architecture — methodology-rules

_Generated by BMAD document-project (deep scan), 2026-06-23. Single backend part (Nx monorepo)._

## Executive Summary

methodology-rules is a **serverless, function-per-rule** validation engine. Each rule is an independent AWS Lambda that validates one aspect of a BOLD methodology document and reports an audit verdict to the Smaug API. There is no shared runtime state between rules; the monorepo exists to share types, document-access code, and test infrastructure — not runtime.

## Technology Stack

See [project-overview.md](./project-overview.md#tech-stack-summary). Core: TypeScript 5.9.3 (strict) on Node 24.14.1, Nx 22.6.1, Vitest 4.1.0, Zod 4.3.6, AWS Lambda.

## Architecture Pattern

**Layered + function-per-rule.** Nx tag boundaries enforce a strict dependency direction:

```
apps/methodologies/*            deployable units (bundle one rule each)
        │  depends on
        ▼
libs/.../rule-processors/*      validation logic (RuleDataProcessor subclasses)
        │  depends on
        ▼
libs/shared/*                   contract, lambda wrapper, document access, helpers, testing
```

- Processors depend on `libs/shared/*` **only** — never on other processors.
- Shared libraries are independent and do not depend on processors or apps.
- Cross-library imports use `@carrot-fndn/*` path aliases, never relative boundary-crossing paths.

## Processing Pipeline

```
RuleInput (S3 event reference: documentId, documentKeyPrefix, responseUrl, responseToken, requestId)
  → Lambda handler  (wrapRuleIntoLambdaHandler)
      → RuleInputSchema.parse(event)            # Zod validation at the boundary
      → processor.process(ruleInput)            # RuleDataProcessor.process
          → DocumentQueryService                # traverse S3 document graph for related docs
              → CachedDocumentLoaderService     # per-invocation cache of S3 fetches
          → validation logic                    # BigNumber / geolib / Zod / Textract
          → mapToRuleOutput → PASSED | FAILED | REVIEW_REQUIRED
      → toUpstreamCompatibleOutput              # REVIEW_REQUIRED → FAILED (until Smaug supports it)
      → reportRuleResults(ruleOutput)           # SigV4-signed POST to Smaug AUDIT_URL
  ← Sentry-wrapped handler; unrecoverable errors propagate to the Lambda runtime (retry/DLQ)
```

## Key Components

### Rule contract (`libs/shared/rule/types`)
- `RuleInput` — `documentId`, `documentKeyPrefix`, `requestId`, `responseToken`, `responseUrl`, optional `environment`, `parentDocumentId`, `ruleName`.
- `RuleOutput` — `resultStatus` (`PASSED`/`FAILED`/`REVIEW_REQUIRED`), optional `resultComment`, `resultContent`, plus the response routing fields echoed back.
- `IRuleDataProcessor` — `process(data: RuleInput): Promise<RuleOutput>`.

### Processor base classes
- `RuleDataProcessor` (abstract, `libs/shared/app/types`) — the root contract; implements `process()`.
- `ParentDocumentRuleProcessor<RuleSubject>` (`libs/shared/methodologies/bold/processors`) — the common BOLD base; subclasses implement `evaluateResult()` and the framework handles document loading, mapping, and result shaping.

### Lambda wrapper (`libs/shared/lambda/wrapper`)
`wrapRuleIntoLambdaHandler(rule)` is the single entry-point boilerplate: initializes Sentry, parses the event into `RuleInput` (Zod), invokes `rule.process()`, reports the result, sets Sentry tags on failure, and lets errors propagate. Rule Lambdas are therefore one line: `export const handler = wrapRuleIntoLambdaHandler(new XProcessor())`.

### Document access (`libs/shared/document`, `libs/shared/methodologies/bold`)
- `DocumentQueryService` traverses the document relationship graph in S3 to find related documents (e.g. a MassID's events, manifests, certificates).
- `CachedDocumentLoaderService` caches S3 fetches within a single invocation.
- BOLD matchers/types model document events (`BoldDocumentEventName`) and entities.

### Result reporting (`libs/shared/rule/result`)
`reportRuleResults` signs the request with SigV4 (assuming `SMAUG_API_GATEWAY_ASSUME_ROLE_ARN` via STS) and POSTs the verdict to the Smaug `AUDIT_URL`.

### Document extraction (`libs/shared/document-extractor*`, `text-extractor`)
Field-neutral `DocumentParser<T>` framework. Specialized extractors (scale-ticket, transport-manifest, recycling-manifest) use Textract OCR. Parsers return optional fields (`undefined` when absent) and set `reviewRequired` by extraction confidence; required-field decisions belong to rule processors.

## Data Architecture

No database. State lives in **S3** (methodology execution documents + attachments). Rules read documents and emit verdicts; they never write documents. See [data-models.md](./data-models.md).

## Cross-Cutting Concerns

| Concern | Approach |
|---|---|
| Validation | Zod at every boundary (`.parse` / `.safeParse`); types via `z.infer`. |
| Numerics | `bignumber.js`, 10 dp, ROUND_DOWN — never raw JS float math. |
| Geo | `geolib` for distance / polygon containment. |
| Logging | `pino` structured logs (document/request IDs); never `console.log`; never log PII. |
| Errors | Enrich + propagate; let the Lambda runtime retry/DLQ; no silent catches. |
| Observability | Sentry (`@sentry/aws-serverless`) + optional CloudWatch metrics. |
| Auth (egress) | STS AssumeRole + SigV4 to call Smaug. |

## Testing Strategy

Vitest. Each processor ships a unit spec (`*.processor.spec.ts`), a Lambda E2E spec (`*.lambda.e2e.spec.ts`, mandatory), and shared `*.test-cases.ts` reused by both. Prefer `stubRuleInput()` / `createStubFromSchema()` and `@carrot-fndn/shared/testing` stubs. See [development-guide.md](./development-guide.md).

## Deployment Architecture

Per-rule webpack bundles, uploaded to S3 by Nx-affected CI per environment (development/production), us-east-1. See [deployment-guide.md](./deployment-guide.md).
