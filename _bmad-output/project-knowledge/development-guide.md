# Development Guide — methodology-rules

_Generated by BMAD document-project (deep scan), 2026-06-23._

## Prerequisites

- **Node** 24.14.1 (`.nvmrc` — use `nvm use`)
- **pnpm** 10.18.3 (enforced by the `preinstall` script — npm/yarn are blocked)
- AWS credentials only needed for deploy/remote-cache, not for local tests

```bash
nvm use
pnpm install
```

## Everyday Commands

All targets run through Nx. Scope to one project, or use the affected/all variants.

| Task | Single project | Affected | All |
|---|---|---|---|
| Test (Vitest) | `pnpm test <project>` | `pnpm test:affected` | `pnpm test:all` |
| Lint (+fix) | `pnpm lint <project>` | `pnpm lint:affected` | `pnpm lint:all` |
| Type-check | `pnpm ts <project>` | `pnpm ts:affected` | `pnpm ts:all` |
| Build Lambda | `pnpm build-lambda <project>` | `pnpm build-lambda:affected` | `pnpm build-lambda:all` |

Run a single test file: `pnpm nx test <project> --testFile=<path>`.

## Scaffolding a New Rule (do NOT hand-create files)

```bash
pnpm create-rule <name> <scope> <description>          # generates the full processor project
pnpm apply-methodology-rule <methodology> <rule> <scope> # wires it into a methodology rule set
```

Generated processor layout (`libs/methodologies/{methodology}/rule-processors/{type}/{name}/src/`):

```
{name}.lambda.ts            # export const handler = wrapRuleIntoLambdaHandler(new {Name}Processor())
{name}.processor.ts         # extends ParentDocumentRuleProcessor<RuleSubject>; implements evaluateResult()
{name}.rule-definition.ts   # ruleDefinition satisfies BaseRuleDefinition
{name}.constants.ts         # result-comment templates + thresholds
{name}.helpers.ts           # stateless arrow-const helpers (not private methods)
{name}.errors.ts {name}.types.ts {name}.validators.ts (optional)
{name}.processor.spec.ts    # unit
{name}.lambda.e2e.spec.ts   # E2E (mandatory)
{name}.test-cases.ts        # shared test data (used by unit + e2e)
index.ts  project.json  vitest.config.ts
```

## Implementation Conventions (must-follow)

- Processor extends `ParentDocumentRuleProcessor<RuleSubject>` and implements `evaluateResult()`. Business logic lives in the processor, **never** in the Lambda file.
- Import shared code only via `@carrot-fndn/*` path aliases (`tsconfig.paths.json`). No relative paths crossing Nx boundaries; no cross-processor imports.
- Strict TypeScript: explicit return types on exports, named exports only, no `any`/`!`/`@ts-ignore`. Narrow indexed access (`noUncheckedIndexedAccess`). Infer types from Zod.
- Numeric comparisons via `bignumber.js` (10 dp, ROUND_DOWN). Geo via `geolib`.
- Validate boundary inputs with Zod; log with `pino` (never `console.log`, never PII).
- Full detail: `.ai/rules/*.md` and [../project-context.md](../project-context.md).

## Testing

- **Vitest** (not Jest). Specs are `*.spec.ts` colocated with source.
- Every processor needs a unit spec **and** a `*.lambda.e2e.spec.ts` (mandatory).
- Define test data once in `{name}.test-cases.ts`; share it between unit and e2e.
- Prefer `stubRuleInput()` / `createStubFromSchema()` and stubs from `@carrot-fndn/shared/testing` over concrete values.
- No real data — synthetic PII, company names, tax IDs only.
- AWS SDK mocked with `aws-sdk-client-mock`.

## Codegen — keep generated artifacts in sync

After editing a methodology's `rules.config.ts`, run all four (any order):

```bash
pnpm generate:readmes
pnpm generate:manifest
pnpm generate:methodology-readmes
pnpm generate:methodology-framework-rules
```

Never hand-edit `codecov.yml`, per-rule READMEs, or methodology-application READMEs (generated, marked "do not edit"). If `generate:readmes` errors with `ENOTDIR … .DS_Store`, delete stray `.DS_Store` files under `apps/ libs/ tools/` and rerun.

After editing anything under `.ai/`, run `pnpm ai:sync` then `pnpm ai:check` (keeps Cursor/Claude/Codex adapters in parity).

## Local Tooling

```bash
pnpm run-rule -- <args>          # execute a rule locally (rule-runner-cli)
pnpm extract-document -- <args>  # run a document extractor (document-extractor-cli)
```

## Before Opening a PR

```bash
pnpm lint:affected
pnpm ts:affected
pnpm test:affected
```

Commits follow Conventional Commits `<type>(<scope>): <description>` — scopes `nx | rule | shared | script`, imperative + lowercase, header ≤ 100 chars. PRs via `gh` CLI with `-a @me -r @carrot-foundation/developers -R carrot-foundation/methodology-rules` and a type label.
