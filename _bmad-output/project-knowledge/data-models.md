# Data Models — methodology-rules

_Generated by BMAD document-project (deep scan), 2026-06-23._

There is **no database**. Persistent state lives in **S3** as methodology execution documents (and attachments such as scale tickets and manifests). The "data models" here are the TypeScript/Zod types that describe those documents and the rule contract. All runtime types are defined as Zod schemas and inferred with `z.infer` — the schema is the single source of truth.

## Storage

| Store | Env var | Contents |
|---|---|---|
| Document bucket | `DOCUMENT_BUCKET_NAME` | Methodology execution documents (the document graph) |
| Attachment bucket | `DOCUMENT_ATTACHMENT_BUCKET_NAME` | Scale tickets, transport/recycling manifests, etc. |

Documents reference each other, forming a graph that `DocumentQueryService` traverses (cached per invocation by `CachedDocumentLoaderService`).

## Core Contract Types (`libs/shared/rule/types`)

- `RuleInput` / `RuleOutput` — see [api-contracts.md](./api-contracts.md).
- `RuleOutputStatus` — `PASSED` | `FAILED` | `REVIEW_REQUIRED` (Zod enum with declaration-merged const for dot-notation access).
- `RuleEnvironment` — `DEVELOPMENT` | `PRODUCTION`.
- `BaseRuleDefinition` — rule metadata shape (`name`, `slug`, `version`, `events`, `description`).

## Branded / shared types (`libs/shared/types`)

- `DocumentId` — branded document identifier (`DocumentIdSchema`).
- `NonEmptyString` — used for layout identifiers in the document-extractor framework.
- Generic cross-cutting types (e.g. `MeasurementUnit`) live here, not in domain-scoped libraries.

## BOLD methodology types (`libs/shared/methodologies/bold/types`)

Domain-scoped types carry a **`Bold`** prefix (naming-convention rule), e.g.:

- `BoldDocumentEventName` — enumerates document events that trigger rules (e.g. `SORTING`, drop-off, weighing, …). A rule's `ruleDefinition.events` references these.
- BOLD document categories, participant/accreditation types, vehicle types, etc. — prefixed to avoid collision with generic concepts (`BoldVehicleType`, not `DocumentEventVehicleType`).

> Naming rule: declare types/schemas/enums in a domain-scoped library **with the domain prefix**; put truly generic types in `libs/shared/types/`.

## Document Extractor models (`libs/shared/document-extractor*`)

The extraction framework is **field-neutral**:

- `DocumentParser<T>` — all extracted fields are **optional**; a parser returns `undefined` (never `""`) for fields it cannot extract.
- `reviewRequired` — set by **extraction confidence** (layout-match score < 0.5), not by business rules.
- `layoutId` / `layouts` — typed as `NonEmptyString`.
- Default layouts live in `defaults.ts`, added only for stable, well-known document formats.
- Required-field decisions are delegated to **rule processors**, never enforced in parsers.

Specialized extractors:

| Library | Document | Tech |
|---|---|---|
| `document-extractor-scale-ticket` | Scale tickets | Textract OCR |
| `document-extractor-transport-manifest` | Transport manifests | Textract OCR |
| `document-extractor-recycling-manifest` | Recycling manifests | Textract OCR + pdf-lib |

## Methodology rule-set config (`rules.config.ts`)

Each methodology framework defines its rule set in a `rules.config.ts`:

- `libs/methodologies/bold-carbon/rules/src/rules.config.ts`
- `libs/methodologies/bold-recycling/rules/src/rules.config.ts`

> **Generated-artifact rule:** after editing any `rules.config.ts`, regenerate manifests/READMEs/codecov flags via `pnpm generate:readmes`, `generate:manifest`, `generate:methodology-readmes`, `generate:methodology-framework-rules`. Never hand-edit `codecov.yml` or generated READMEs.

## Numeric representation

Quantities (weights, factors, emissions) are compared with **`bignumber.js`** at 10 decimal places, ROUND_DOWN — never JS `number` arithmetic — to avoid floating-point drift in audit verdicts.
